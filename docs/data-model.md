# Data model & processing flow

This document describes the Firestore schema, message format, and processing lifecycle for the SMMUN registration system.

---

## Collection: `submissions`

Each document represents a single registration submission (`delegacion` or `faculty`).

---

## Document ID

- `submission_id` (string, autogenerated)

Used as:
- primary identifier across the system
- reference in Pub/Sub messages
- correlation key in logs

---

## Top-level fields

- `type`: string  
  - `"delegacion"` | `"faculty"`

- `status`: string  
  - `"pending"` | `"processing"` | `"completed"` | `"failed"`

- `created_at`: timestamp (UTC)  
- `updated_at`: timestamp (UTC)

- `file_path`: string  
  - Cloud Storage path  
  - Example:  
    ```
    uploads/delegaciones/DELEGACION_{nombre}_{id}.pdf
    ```

- `error_message`: string (optional)  
  - Present only if `status = "failed"`

---

## Field: `data` (object)

Contains all form-specific data. Structure depends on `type`.

---

# Delegación / codelegación

## Structure

```json
{
  "modalidad": "individual" | "pareja",

  "delegacion_oficial": {
    "is_oficial": boolean,
    "nombre": string,
    "responsable": string
  },

  "participantes": [
    {
      "nombre": string,
      "apellido": string,
      "edad": number,
      "celular": string,
      "correo": string,
      "pais": string,
      "ciudad_estado": string,
      "escolaridad": string,
      "escuela": string,
      "contacto_emergencia": string,
      "info_extra": string | null
    }
  ],

  "comites": [
    {
      "nombre": string,
      "opciones": [string, string, string | null]
    }
  ]
}
```

---

## Constraints

* `participantes`

  * length = 1 → individual
  * length = 2 → codelegación

* `comites`

  * exactly 3 entries
  * ordered by preference (priority)

# Status lifecycle

Each submission follows a controlled lifecycle:

```text
pending → processing → completed
                    ↘ failed
```

### States

* `pending`

  * Created by API
  * Event published to Pub/Sub

* `processing`

  * Claimed by worker (transactional)
  * Prevents duplicate processing

* `completed`

  * Sheets + email successfully executed

* `failed`

  * Any error during processing
  * `error_message` is recorded

---

## Idempotency

Worker uses Firestore transactions to ensure:

* a submission is processed **at most once**
* duplicate Pub/Sub deliveries are safely ignored

---

# Pub/Sub Message Format

```json
{
  "submission_id": "string",
  "request_id": "string",
  "submission_type": "delegacion" | "faculty"
}
```

### Notes

* `request_id` enables trace correlation across services
* `submission_type` improves observability

---

# Processing flow (worker)

Given `submission_id`:

1. Retrieve document from Firestore
2. Attempt to claim (`status → processing`) using transaction
3. If already processed → exit (idempotent behavior)
4. Generate signed URL for `file_path`
5. Process submission:

   * append data to Google Sheets
   * send email via Resend
6. Update Firestore:

   * `status = "completed"`
   * OR `status = "failed"` + `error_message`

---

# API responsibilities

On form submission:

1. Validate input (including constraints)
2. Upload file to Cloud Storage
3. Create Firestore document:

   * `status = "pending"`
4. Publish Pub/Sub message
5. Return HTTP 303 redirect (confirmation page)

---

# Observability

* Each request is assigned a `request_id`
* Propagated:

  ```
  API → Pub/Sub → Worker
  ```
* Used for:

  * structured logging
  * trace grouping in GCP

---

# Security notes

* Files are stored privately in Cloud Storage
* Access is provided via **signed URLs** (temporary, time-limited)
* No public bucket exposure
